# Backup — Docker Compose Files

## Principle

Docker compose files are versioned in a dedicated Git repository, separate from application data.

```
Git repo (compose files)  ←→  /opt/docker/docker-compose/
Application data (appdata)     /opt/docker/appdata/          ← NOT in Git
```

## What is in the Git repo

- All `docker-compose.yml` files
- Environment variable files (`.env`) — with secrets either excluded or replaced
- Labels, network definitions, volume mappings

## What is NOT in the Git repo

- `/opt/docker/appdata/` — runtime data generated by containers (databases, config files written by apps, etc.)
- This data is covered by **Timeshift only** (see below)

## appdata backup coverage

| Tool | Covers appdata | Notes |
|---|---|---|
| Timeshift | ✅ | System-wide snapshot — includes `/opt/docker/appdata/` |
| Borg | ❌ | Only backs up specific user data directories |
| Git | ❌ | Source files only, not runtime data |

⚠️ **Implication**: if the system drive fails and the latest Timeshift snapshot is outdated, appdata changes since the last snapshot may be lost. This is an accepted trade-off.

## Restoring a service

To bring a service back up:
1. Clone the compose repo
2. Restore appdata from Timeshift (or accept data loss if snapshot is old)
3. Run `docker compose up -d`

See [Docker restoration](../restoration/docker-restore.md) for details.
